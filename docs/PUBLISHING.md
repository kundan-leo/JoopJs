## Repository Structure for Publishing

```
joopjs/
├── src/                    ← TypeScript source (never published)
├── tests/                  ← Vitest tests (never published)
├── dist/                   ← Built output (generated by tsup)
│   ├── index.cjs
│   ├── index.mjs
│   ├── index.d.ts
│   └── ...sub-paths...
├── prod/                   ← ✅ PUBLISH WORKSPACE
│   ├── package.json        ← Clean manifest (no devDeps, no build scripts)
│   ├── dist/               ← Copied from root dist/ by publish scripts
│   ├── CHANGELOG.md        ← Copied from root
│   └── .npmignore
├── playground/             ← Dev UI (never published)
├── scripts/                ← Release helpers (never published)
├── publish.bat             ← Windows publish script
├── publish.sh              ← Linux/macOS publish script
├── package.json            ← Root manifest (includes devDependencies)
├── tsup.config.ts          ← Build config
└── CHANGELOG.md            ← Source of truth for changelog
```

What actually gets uploaded to npm (controlled by `prod/package.json` `"files"` field):
- `dist/**` — all compiled JS, CJS, ESM, declaration files and source maps
- `CHANGELOG.md`

## First-Time Setup

### 1. Clone the repository

```bash
git clone https://github.com/your-org/joopjs.git
cd joopjs
```

### 2. Install dependencies

```bash
npm install
```

### 3. Verify the test suite passes

```bash
npx vitest run
# → 143 files, 2168 tests, 0 failures
```

### 4. Do a dry-run build

```bash
npx tsup
# → dist/ is populated
```

### 5. Inspect what would be published

```bash
cd prod
npm pack --dry-run
cd ..
```

You should see only `dist/` and `CHANGELOG.md` in the file list.

## Versioning Policy (SemVer)

JoopJS follows [Semantic Versioning](https://semver.org/) — `MAJOR.MINOR.PATCH`.

| Bump | When | Example |
|---|---|---|
| `patch` | Bug fix, docs update, internal refactor — no API change | `2.0.0 → 2.0.1` |
| `minor` | New service, new method, new sub-path export — backward-compatible | `2.0.0 → 2.1.0` |
| `major` | Breaking change: rename/remove method, change param shape, drop sub-path | `2.0.0 → 3.0.0` |

**Rules:**
- Never publish without bumping the version.
- Versions are bumped in both `package.json` (root) **and** `prod/package.json` by the publish scripts automatically.
- Never manually edit the `version` field in `prod/package.json` — the scripts keep them in sync.
- Once a version is published, it cannot be overwritten (npm enforces this).

## The `prod/` Workspace

The `prod/` folder is the **npm publish workspace** — a clean, stripped copy of the package containing only what end-users receive.

Key differences from the root `package.json`:

| Field | Root `package.json` | `prod/package.json` |
|---|---|---|
| `devDependencies` | Present | **Absent** |
| `scripts` | build, test, playground… | **Absent** |
| `dependencies` | Runtime deps | Same runtime deps |
| `exports` | Full 35 sub-paths | Same 35 sub-paths |
| `sideEffects` | — | `false` (tree-shaking signal) |

**Never run `npm publish` from the project root** — the root `package.json` has `devDependencies` that would bloat the published package.

Always publish from `prod/`:
```bash
cd prod
npm publish --access public
```

## Publishing — Windows (.bat)

Open **Command Prompt** or **PowerShell** in the project root:

```batch
:: Publish at current version (no bump)
publish.bat

:: Bump patch version (2.0.0 → 2.0.1) then publish
publish.bat patch

:: Bump minor version (2.0.0 → 2.1.0) then publish
publish.bat minor

:: Bump major version (2.0.0 → 3.0.0) then publish
publish.bat major
```

The script runs these steps automatically:
1. Checks Node.js and npm are installed
2. Runs the full test suite (`vitest run`)
3. TypeScript strict check (`tsc --noEmit`)
4. Builds with tsup (`dist/` is regenerated)
5. Bumps version in both `package.json` files (if a bump arg was passed)
6. Copies `dist/` and `CHANGELOG.md` into `prod/`
7. `npm pack --dry-run` — shows the tarball contents
8. `npm publish --access public` from `prod/`
9. Prints the npm URL and git tagging commands

**If any step fails, the script exits immediately — nothing is published.**

## Publishing — Linux / macOS (.sh)

Make the script executable once:
```bash
chmod +x publish.sh
```

Then run:
```bash
./publish.sh           # publish at current version
./publish.sh patch     # 2.0.0 → 2.0.1 then publish
./publish.sh minor     # 2.0.0 → 2.1.0 then publish
./publish.sh major     # 2.0.0 → 3.0.0 then publish
```

The `.sh` script is identical in behaviour to the `.bat` script.

## Step-by-Step Manual Publish

If you prefer not to use the automated scripts, here is the full manual sequence:

```bash
# 1. Run all tests
npx vitest run

# 2. TypeScript check
npx tsc --noEmit

# 3. Build
npx tsup

# 4. Bump version manually in BOTH files
#    (or use: npm version patch --no-git-tag-version)
#    Edit "version" in package.json and prod/package.json to match

# 5. Copy artefacts into prod/
rm -rf prod/dist
cp -r  dist prod/dist
cp     CHANGELOG.md prod/CHANGELOG.md
cp     README.md    prod/README.md   # optional

# 6. Inspect the tarball
cd prod
npm pack --dry-run

# 7. Publish
npm publish --access public
cd ..

# 8. Commit and tag
git add package.json prod/package.json CHANGELOG.md
git commit -m "chore: release v2.0.1"
git tag v2.0.1
git push && git push --tags
```

## Git Workflow

### Branch strategy

```
main            ← production-ready, tagged releases
└── develop     ← integration branch
    └── feat/*  ← feature branches
    └── fix/*   ← bug fix branches
    └── chore/* ← tooling, docs, refactors
```

### Standard release flow

```bash
# 1. Work on a feature branch
git checkout -b feat/new-savings-goal develop

# ... make changes, write tests ...

# 2. Merge back to develop
git checkout develop
git merge --no-ff feat/new-savings-goal
git branch -d feat/new-savings-goal

# 3. Merge develop → main when ready to release
git checkout main
git merge --no-ff develop

# 4. Run publish script (bumps version, builds, publishes)
./publish.sh minor          # or publish.bat minor on Windows

# 5. Tag the release
git tag v2.1.0
git push && git push --tags
```

### Commit message conventions

```
feat: add JoopGoalSavingsService                  ← new feature (triggers minor bump)
fix:  correct EMI calculation for zero-rate loans ← bug fix (triggers patch bump)
docs: update sanctions screening examples          ← docs only
chore: upgrade tsup to 8.4.0                       ← tooling (no bump needed)
refactor: extract shared crypto helpers            ← internal (no bump needed)
BREAKING CHANGE: rename JoopBudget → JoopBudgetService ← (triggers major bump)
```

### Tagging a release

```bash
# Annotated tag (preferred — visible on GitHub releases page)
git tag -a v2.1.0 -m "Release v2.1.0 — New savings goal and portfolio services"

# Lightweight tag (quick reference)
git tag v2.1.0

# Push the tag to remote
git push origin v2.1.0
# or push all tags
git push --tags
```

### Viewing tags

```bash
git tag --list "v*"            # list all version tags
git show v2.0.0                # inspect a specific tag
git log v2.0.0..HEAD --oneline # commits since v2.0.0
```

## Post-Publish Checklist

After a successful `npm publish`, do these steps:

```bash
# 1. Verify it is live on npm
npm view joopjs version
# → 2.1.0

# 2. Test install in a clean directory
mkdir /tmp/test-joopjs && cd /tmp/test-joopjs
npm init -y
npm install joopjs
node -e "const { JoopGcmService } = require('joopjs'); console.log(new JoopGcmService())"
cd -

# 3. Commit the version bump (if not done by the script)
git add package.json prod/package.json CHANGELOG.md
git commit -m "chore: release v2.1.0"

# 4. Create and push the git tag
git tag v2.1.0
git push && git push --tags

# 5. Update CHANGELOG.md with release notes
#    (the publish script creates a placeholder entry — fill in the details)

# 6. Announce (Slack / team channel / GitHub release)
```

## Hotfix / Patch Release

When a critical bug is found in a published version:

```bash
# 1. Branch from the tag (not from develop)
git checkout -b fix/sanctions-match-bug v2.1.0

# 2. Fix the bug + add a regression test
#    ... edit src/security/sanctions-screening.service.ts ...
#    ... add test to tests/sanctions-screening.test.ts ...

# 3. Verify the fix
npx vitest run
npx tsc --noEmit

# 4. Publish the patch
./publish.sh patch             # bumps 2.1.0 → 2.1.1

# 5. Tag and merge back
git tag v2.1.1
git push && git push --tags
git checkout develop && git merge fix/sanctions-match-bug
git branch -d fix/sanctions-match-bug
```

## Dry Run (No Publish)

To run through all steps (tests, build, version bump, tarball inspection) **without** actually publishing:

```bash
# Using the existing release script in dry-run mode
node scripts/release.mjs --dry-run

# Or manually — go through all steps but skip the npm publish
./publish.sh patch
# When the script reaches "npm publish --access public", Ctrl+C to abort
# (not ideal — prefer release.mjs --dry-run for a clean dry run)
```

## Unpublishing / Deprecating

npm allows unpublishing within **72 hours** of publishing:

```bash
# Unpublish a specific version (within 72h only)
npm unpublish joopjs@2.1.0

# Unpublish the entire package (within 72h only, use with extreme caution)
npm unpublish joopjs --force
```

After 72 hours, unpublishing is blocked. Use **deprecation** instead:

```bash
# Deprecate a specific version
npm deprecate joopjs@2.1.0 "Critical bug — please upgrade to 2.1.1"

# Deprecate all 2.x versions
npm deprecate "joopjs@2.x" "Please upgrade to v3"

# Un-deprecate
npm deprecate joopjs@2.1.0 ""
```

## Troubleshooting

### `npm publish` fails with 403 Forbidden

```
npm ERR! 403 Forbidden - PUT https://registry.npmjs.org/joopjs
```

**Causes and fixes:**
- Not logged in → `npm login`
- Package name already taken by another user → rename the package
- 2FA required → `npm publish --otp=<your-6-digit-code>`
- Scope issue → ensure `--access public` for unscoped packages on the free tier

---

### `npm publish` fails with 409 Conflict

```
npm ERR! 409 Conflict - PUT ... - You cannot publish over the previously published versions
```

The version number already exists on npm. You must bump the version:
```bash
# In both package.json and prod/package.json
"version": "2.1.1"   # was "2.1.0"
```

---

### Tests fail during publish

The publish script runs `vitest run` as step 2. If tests fail:
1. The script aborts — **nothing is published**
2. Fix the failing tests
3. Re-run the publish script

Never skip the test step.

---

### TypeScript errors during publish

The script runs `tsc --noEmit` as step 3. Fix all type errors before publishing:
```bash
npx tsc --noEmit
# Inspect errors, fix them
# Re-run publish script
```

---

### `dist/` is empty or outdated

If `dist/` exists from a previous build but source has changed:
```bash
npx tsup            # full clean rebuild (tsup has clean: true in config)
ls dist/            # should show index.cjs, index.mjs, index.d.ts
```

---

### Wrong version published

If you notice a version mismatch after publishing:
1. Deprecate the wrong version: `npm deprecate joopjs@2.1.0 "Published in error — use 2.1.1"`
2. Fix the version number
3. Publish the correct version

## npm Free Tier Limits

The free npm account supports:
- **Unlimited public packages** — `joopjs` is public, no restrictions
- **`--access public`** is required for unscoped packages on free accounts
- Scoped packages (`@kundan/joopjs`) are private by default on free tier — use `--access public` to make them public

```bash
# Free tier publish command
npm publish --access public

# Scoped package on free tier — must add --access public
npm publish --access public    # (@scope/package on free tier)
```

Private packages require the **npm Pro** plan ($7/month).

For a public SDK like JoopJS, the free tier is all you need.
